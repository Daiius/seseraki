### 時制方針: 「あるべき設計の理想形」を基準にする（2026-07-16）
- PRD は現実装の記録に留めず、**あるべき設計の理想形を正**として書く。
- 現実装が理想と異なる箇所は、理想を本文に書いた上で**現状との gap** として明示・管理する。
- 帰結: 現 DB スキーマ・アーキが理想でない場合、PRD は理想を記述し、実装追従は gap 項目にする。
- 前提は変わらず**個人用・ローカル運用**。理想は「個人用棋譜解析ツールとしての理想」であって
  エンタープライズ化ではない（過剰設計を避ける）。

### 解析来歴（engine/eval）は持たない（2026-07-16）
- **決定**: 解析エンジン・評価関数の識別子（MATERIAL/NNUE・評価関数名）を DB に**記録しない**。
- 理由: 本番は NNUE 単一運用で開発 MATERIAL の数値は一時的な確認用。異常な評価値は目視で気づける。
  エンジン側に自己申告を仕込むのは手間で、個人用途には**過剰設計**。
- 保持するのは `candidateMoves.depth`（候補手単位。既存）のみ。
- **帰結**: 局面単位の再解析は depth 別の複数世代を持たず、**最新 1 世代に上書き**（単一エンジン前提）。

### LLM 解説の生成アーキ（2026-07-16）
- **決定**: プロンプト（解析結果込み Markdown）生成を **server のエンドポイント化**（例 `GET /kifus/:id/commentary-prompt`）。
- **決定（対象範囲）**: **手動トリガーのみ**（解析済み全棋譜への自動付与はしない）。Web の「解説生成」ボタンで
  キュー投入 → commentator が拾って生成。欲しいものだけ生成し LLM コストを制御下に置く。
- **commentator = 軽量な監視スクリプト**（独立 docker container・worker 側デスクトップに配置）:
  **キュー（`commentaries.status = 'queued'`）を polling** → プロンプトを server から fetch → LLM CLI に流す →
  解説を `commentaries` へ書き戻す。commentator 自身は将棋ロジックを持たず薄く保つ。サブスク内で完結し API 従量課金なし。
- web の「LLM プロンプトをコピー」ボタンも同じプロンプトエンドポイントを使い、書式を単一真実にする。

### shared パッケージを作る（旧「作らない」を改定）（2026-07-16）
- **決定**: `packages/shared` を新設する。
- **責務（凝集。junk drawer にしない）**:
  1. 将棋ドメインの**純ロジック**（board 盤面追跡・USI 変換・USI→日本語・悪手判定・kifu-export プロンプト生成）。
  2. **zod 検証スキーマ**（runtime バリデーション）。
- **旧判断の再評価**: 「shared を作らない」は元々「Hono RPC の型共有を使わず zod 定義を shared に溜め込む」
  傾向への予防だった。しかし (1) 対話盤面ロジックを web と server（プロンプト生成）で共有する必要が生じ、
  (2) **型共有だけでは動作時に不正データを弾けず、runtime 検証には zod の実体が要る**。→ shared の意義が明確化。
- **維持する原則**: **API レスポンス/リクエストの型共有は引き続き Hono RPC**。shared に API 型を重複させない。
- **帰結**: board/usi/kifu-export を web→shared へ移設。server は shared の kifu-export でプロンプト生成
  エンドポイントを提供。commentator は薄いまま。web は対話盤面で shared を import。
- KIF/CSA パーサーは現状 server 専用（消費者が server のみ）なので shared に移さない（第2の消費者が出たら再検討）。

### commentaries テーブル（2026-07-16）
- **決定**: `commentaries` は kifu と **1:1**、再生成は**同行を上書き**（解析来歴非保持の方針と揃える）。
- **キューを兼ねる**（別途キューテーブルは持たない。薄い watcher 思想）: `status`(queued/done/failed) を持ち、
  「解説生成」ボタンで `queued` 行を作る/戻す → commentator が `queued` を polling → 生成後 `done`。
  失敗は `failed` + `error` で受け、無限リトライを防ぐ（poison-pill と同じ思想）。
- カラム: `kifuId`(PK/FK), `status`(enum), `body`(text?, 完了まで null), `llmModel`(varchar?),
  `promptVersion`(varchar?), `error`(text?, failed 用), `createdAt` / `updatedAt`。
- Web の「解説あり」表示は `status = 'done'` で判定。

### 入力形式のスコープ（2026-07-16）
- **決定**: Web 手動投入は **KIF 貼り付け（実装済み）+ CSA 直接貼り付け**（既存の CSA→KIF 変換器を再利用）。
- **対象外**: KIF ファイルアップロード / SFEN（単一局面で棋譜解析のスコープ外）。
- CSA→KIF 変換器は現状は取り込み専用ディレクトリ配下だが、CSA 貼り付けは一括取り込みと無関係なので、
  一般的な変換として server 内の中立な場所に置く（消費者は server のみなので shared には移さない）。

### ポイズンピル対策（解析できない棋譜）（2026-07-16）
- **主軸 = worker 側の失敗状態**: kifus に `analysisError`(nullable) を追加。worker はエンジンの
  異常終了/illegal move を検知したら当該 kifu に記録し、**エンジンを再起動して次へ**進む。
  poll は「`analysisCompletedAt IS NULL` かつ `analysisError IS NULL` の最古」を引く。
  → ポイズンピルはスキップされ**キューが詰まらない**。UI に「解析失敗」バッジ、再試行は error クリア。
- **投入時検証は best-effort（保証ではない）**: 完全な合法性検証は将棋ルール一式（打ち歩詰め等）の
  実装が要り容易でない。`applyMove` が破綻しないか程度の軽い整合チェックに留め、取りこぼしは worker 側で受ける。
- 具体的なバグの症状・再現・修正手順といった**作業メモは PRD に書かず** `.claude-personal/` に置く。

### KIF 変換の堅牢化・メタ抽出・失敗時動作（2026-07-17）

実棋譜（`８四全` を含む 70 手）で「記録は入るが解析エンジンが illegal move で異常終了し、以後同じ棋譜を
無限リトライしてキューが停止する」不具合を起点に、入力面と失敗時動作をまとめて詰めた。

- **原因**: KIF パーサが成銀の一文字略記 `全`（同様に `圭`=成桂 / `杏`=成香）を未対応で手 57 が黙って欠落 →
  1 手欠落で手番 parity がズレ、構文上妥当な USI 列のまま盤面が破綻 → illegal move。加えて `kifToUsiMoves` が
  パースエラーを無視して部分的 `usiMoves` を保存していた。
- **入力側の決定**:
  - パースエラー（指し手行の変換失敗）または**非平手手合割**なら `usiMoves = null` で保存（記録・メタは残す）。
    壊れた部分 `usiMoves` を worker に渡さないのが要点。完全合法性チェック（`shared` + `applyMove`）は将来の理想として残し、
    当面はこの軽量ガード＋ worker 失敗状態で受ける（best-effort の具体化）。
  - 成駒略記対応・終局マーカー認識（`詰み` 等・最初のマーカーで指し手収集を打ち切り＝末尾の分岐ブロック混入防止）。
  - メタ抽出（sente/gote/dan/playedAt）。**result は終局マーカー＋手番 parity から swars 互換コードを導出**し、
    既存の勝敗バッジ・LLM エクスポートを無改修で活かす。
  - **タイトル任意化・自動生成**（空なら `${sente} vs ${gote}`）。旧「KIF はユーザー入力」を改定（KIF インポート常用化のため）。
- **失敗時動作の決定**:
  - 失敗分類: 棋譜起因（illegal / エンジン異常終了 / **timeout**）＝恒久（`analysisError` 記録 → poll 除外）、
    インフラ起因（submit 失敗）＝一時（再試行）。恒久時は**エンジン無条件再起動**。
  - `analyze()` はプロセス `close` で即 reject（60 秒ハング解消）。
  - 再試行 UI アクション＝**`POST /kifus/:id/reanalyze`**: `kifText` 再変換で `usiMoves`・メタを作り直し
    `analysisError`/`analysisCompletedAt` をクリアして再キュー（`title`/`memo` は温存）。error クリアだけだと
    壊れた `usiMoves` のまま再度落ちるため、再変換で既存の壊れ棋譜も復旧する。
- **スコープ外（今回入れない）**: `shared` 抽出を伴う完全合法性チェック、KIF ファイルアップロード、駒落ちの USI 変換対応。
- 具体的なバグ症状・再現・修正手順の作業メモは `.claude-personal/`（公開 PRD には書かない）。

### 棋譜詳細の「候補手詳細」テーブルを廃止（2026-07-21）

棋譜詳細ページ下部の折り畳み「候補手詳細」（全局面 × 全候補手のテーブル）を**製品仕様として持たない**と決めた
（[05](../05-analysis.md) §2.5 を改訂。issue #26 / PR #36）。

- **理由（運用）**: 実運用でまず開いていない。読み筋を見たい局面は評価値グラフの悪手マーカーから飛べるので、
  その局面の候補手一覧（[05](../05-analysis.md) §2.2・読み筋 + 探索深さ + 実手マーク + 分岐操作）で用が足りる。
- **理由（コスト）**: `<details>` は閉じていても中身がレンダーされる。この表は局面数 × MultiPV 行を生成し、
  各行の読み筋日本語化が手数ぶん `applyMove`（盤面のディープコピー）を回すため、**詳細ページを開いた瞬間に
  走る盤面コピーの最大の発生源**だった。遅延レンダーや仮想化で覆う手もあるが、使っていない機能のために
  実装を増やす判断はしない。
- **[01](../01-domain.md) §1-2「疑問手と分岐を一望したい」との整合**: この動機は評価値グラフ + 悪手マーカー
  （棋譜全体の俯瞰）と、局面ごとの候補手一覧（深掘り）の組み合わせで満たす。テキストで通して読みたい場合は
  LLM 解説用テキスト（[06](../06-llm-commentary.md)）が**全局面の実手と最善評価値**、および**抽出された注目局面の
  候補手・読み筋**を持つ（全局面 × 全候補手ではない点で廃止テーブルとは非等価）。よって 01 の改訂は不要とした。
- **データ側は不変**: `candidateMoves.pv` は盤面直下の読み筋表示・分岐再生・kifu-export が使い続ける
  （悪手判定は `rank`/`move`/`score` だけを見ており `pv` を参照しない）。
  スキーマ・API・worker に変更なし（表示先が 1 つ減るだけ）。

### 棋譜詳細の描画コスト — 折り畳みの遅延マウントと局面の不変・構造共有（2026-07-21）

「候補手詳細」廃止（上記）と React Compiler 導入（issue #25 / PR #35）でも消えない残りの描画コストを
3 点まとめて詰めた（[05](../05-analysis.md) §2 / §2.5 を改訂。issue #27）。

- **折り畳みは開かれるまで中身をマウントしない**。`<details>` は閉じていても中身がレンダーされるため、
  局面評価値テーブル（局面数ぶんの行 × 日本語変換）と LLM 解説用テキスト（`generateKifuMarkdown` が
  全局面を再構築する）が、閉じたまま毎レンダー走っていた。**折り畳みに重いものを入れる構成自体は良い**ので、
  廃止ではなく描画を遅延させる。一度開いた後はマウントを保つ（開閉の往復が重くならず、メモの下書きも消えない）。
- **全局面は 1 度だけ構築する**。ページと盤面コンポーネントで `buildPositions` を二重に呼んでいた。
  React Compiler が効くのはレンダー間のキャッシュであって、**同一レンダー内の重複計算は消えない**ため、
  ページ側で構築して props で配る形にした。
- **`BoardState` は不変・構造共有**。`applyMove` は毎手 81 マス + 持ち駒をディープコピーしていた。
  変化したマスと手番側の持ち駒だけを差し替え、残りの行・駒オブジェクトは共有する。
  代わりに**局面を破壊的に変更してはならない**という契約が生じるので、型定義と [05](../05-analysis.md) §2 に明記した。
  `shared` 抽出（[08](../08-roadmap.md)）で盤面ロジックを移す際もこの契約ごと移す。
- **不正な手の扱いは変えない**（移動元 == 移動先の退化した手で駒が消える等）。合法性検証は `shared` 抽出時の
  課題として据え置き（[04](../04-ingestion.md) §5）、今回は純粋な性能改善に留める。

### 棋譜詳細の削除・再解析は結果を画面に返す（2026-07-21）

削除・再解析の呼び出しは `res.ok` のときだけ遷移 / 再取得しており、**失敗すると何も起きなかった**
（issue #30 の 1 項目目。[05](../05-analysis.md) §2.5 を改訂）。

- **失敗は握り潰さない**。HTTP エラーは `alert-error` にステータスコードを添えて出し、fetch 自体が
  失敗する（サーバー停止・ネットワーク断）ケースは `try/catch` で「サーバーに接続できません」を出す。
  文言と表示手段は既存の [`KifuMemo`](../../packages/web/src/components/KifuMemo.tsx) / 一覧の取り込み
  （[04](../04-ingestion.md)）に揃えた。
- **トーストライブラリは入れない**。通知は現状この 2 操作とメモ保存・取り込みだけで、いずれも
  操作起点の画面に出せば足りる。依存を増やさず daisyUI の `alert` を使う。
- **再解析は成功も通知する**。解析は worker が非同期に拾うため、成功しても画面はほぼ変化しない
  （`analysisError` の消滅くらい）。「開始しました」を出さないと失敗と区別がつかない。
  削除は成功すると一覧へ遷移するので通知しない。
- **実行中はボタンを無効化する**。二重送信を防ぐ。再解析ボタンはドロップダウンと `analysisError`
  アラート内の 2 箇所にあるため、同じフラグで両方を止める。

### 棋譜再生のキーボード操作と、分岐中の ←→ の意味（2026-07-21）

棋譜ビューアの基本操作である `←` / `→` / `Home` / `End` が無く、1 手進めるたびにボタンを狙う必要があった
（issue #30 の 2 項目目。[05](../05-analysis.md) §2.1 を改訂）。

- **分岐中の `←` / `→` は分岐内を移動する**。分岐（候補手の読み筋再生・[05](../05-analysis.md) §2.2）に入っている
  間は、ユーザが今読んでいるのは分岐の局面であって本筋ではない。ここで `→` が本筋を進めると、
  読んでいた分岐が黙って消える。分岐の先頭で `←` すると `branchDepth` が 0 になり自然に本筋へ戻るので、
  「分岐を抜ける」ための別キーは要らない。マウス操作（本筋は上部コントローラー / 分岐は候補手内の ◀▶）とは
  割り当てが異なるが、**キーボードは常に「今見えている手順」を進退させる**方が迷わない。
- **`Home` / `End` は常に本筋の端へ飛ぶ**（分岐は解除）。「最初 / 最後」は棋譜全体に対する位置であって、
  分岐内の端を指す意味づけは無い。上部コントローラーの ⏮ ⏭ と同じ `goToMain` に揃える。
- **リスナは `window` に張る**。盤面はページの主役で、他にフォーカス可能な操作対象が少ないため、
  盤面へのフォーカスを要求すると「クリックしないと効かない」だけの体験になる。代わりに
  **入力欄（`input` / `textarea` / `select` / `contenteditable`）にフォーカスがある間は何もしない**。
  特にコントローラーの range スライダーは `←→` `Home` `End` をブラウザが既定で処理し `onChange` 経由で
  局面が動くため、ここで介入すると 1 回の打鍵で 2 手進む。
- **修飾キー併用（Ctrl / Meta / Alt / Shift）は横取りしない**。ブラウザの戻る/進む・行選択などと衝突するため。
  該当キーを処理したときだけ `preventDefault()` してページのスクロールを止める。
- **discoverability はボタンの `title` に添える**（「戻る (←)」等）。ショートカット一覧の UI は作らない。
- **端での打鍵は何もしない**（レビュー指摘 `OCL-AA77E8E6` を受けて追記）。本筋の先頭で `←`、末尾で `→`、
  既に端にいる `Home`/`End` は局面が動かない。ここで `goToMain` を呼ぶと**開いている候補手 details が
  閉じられ、閲覧中の読み筋が消える**（画面ボタンは同じ境界で `disabled` になるためこの副作用が無い）。
  移動先が現在位置と異なるときだけ遷移し、分岐が残っているときだけ解除のために同位置へも呼ぶ。

### swars 取り込みのページ数を Web から指定する（2026-07-21）

`POST /api/swars/import` は `pages: 1..10` を受け付けるのに、Web は常に `pages: 1` を送っていて
**過去分を遡れなかった**（issue #30 の 3 項目目。[04](../04-ingestion.md) §4 を改訂）。

- **「更新」ボタンの隣にセレクトを置く**。設定画面や環境変数ではなく取り込みの起点に置く。
  ページ数は「今どこまで遡るか」という**その場の判断**で、恒久設定ではないため。
  daisyUI の `join` でボタンと連結し、1 操作の一部であることを見た目でも示す。
- **選択は永続化しない**（リロードで既定の 1 に戻る）。常用は最新 1 ページで、初回セットアップや
  久しぶりの取り込みだけが例外。10 が残ったまま毎回の更新が 10 ページ分の取得を走らせる方が害が大きい。
- **上限 10 は server の zod と二重に持つ**。`MAX_IMPORT_PAGES` に切り出してコメントで対応を明記した。
  検証スキーマの共有は `shared` 抽出（[02](../02-architecture.md) §3.2 / [08](../08-roadmap.md)）待ちの gap。
- **取得済みの重複を気にせず遡ってよい**。`swarsGameKey` の UNIQUE でスキップされる（§4）ので、
  ページ数を多めに指定しても増えるのは履歴取得のリクエストだけ。セレクトの `title` にもその旨を添えた。

### 一覧の日時列は代替表示であることを明示する（2026-07-21）

棋譜一覧の日時列は `playedAt` が無ければ `createdAt` を**無印で**表示していたため、**見えている日時が
対局日時なのか登録日時なのか区別できなかった**（issue #30 の 4 項目目。[05](../05-analysis.md) §2.5 を改訂）。

- **列そのものは分けない**。一覧の並びが `coalesce(playedAt, createdAt)` 降順（[04](../04-ingestion.md) §6）で
  ある以上、この列は「その棋譜を時系列に置く基準値」であって、対局日時と登録日時を別列にすると
  行ごとにどちらが並び順の根拠なのか読み取れなくなる。**1 列のまま、代替のときだけ但し書きを添える**。
- **代替であることは「（登録）」+ 弱色**（`text-base-content/60`）で示す。バッジは状態列（勝敗・解析済み）で
  既に使っており、日時列にも足すと視線の止まる場所が増える。既定は対局日時であり、目立たせたいのは
  「これは例外」という一点だけなので、弱いシグナルで足りる。
- **`title` に理由を添える**（「対局日時が取得できなかったため、登録日時を表示しています」）。
  但し書きだけでは「登録日時も併記されている」と読めるため、**代替であること**を明文化する。
- **メタ抽出を直す話とは切り離す**。`playedAt` が埋まらない棋譜（TZ 欄なし・ヘッダ欠落等）を減らすのは
  取り込み側の課題（[04](../04-ingestion.md) §2）で、それでも 0 にはならない。一覧は残った欠落を
  正直に見せる側に立つ。

### 一覧の検索・絞り込み・並べ替えは server 側で行い、条件は URL に持つ（2026-07-21）

棋譜一覧は 50件ページングだけで、溜まるほど目的の対局に辿り着けなかった（issue #30 の 5 項目目。
[04](../04-ingestion.md) §6.1 / [05](../05-analysis.md) §2.5 を改訂）。

- **絞り込みは server 側（SQL）で行う**。クライアント側で現在ページの 50件だけを絞ると、
  「ヒット件数」も「次のページ」も嘘になる。**件数・ページングと絞り込みは同じ条件で数える**必要がある。
- **条件は URL のクエリに持つ**（コンポーネント state ではなく）。TanStack Router の `validateSearch` /
  `loaderDeps` に載せることで、リロード・戻る/進む・ブックマークが素直に効き、loader の再取得も
  ルーターに任せられる。既定値（`status=all` 等）は URL に載せない。
- **「自分」の定義は web に置いたまま、勝敗の判定材料だけを `self` で server に渡す**。
  自分の名前候補は `VITE_SELF_NAMES` ∪ `VITE_SWARS_USER_ID`（[01](../01-domain.md)）で、
  一覧の勝敗バッジ・棋譜詳細・LLM 解説（[06](../06-llm-commentary.md)）がすべてこれを見ている。
  server にも `SELF_NAMES` を持たせると**同じ設定が二箇所に分かれ、ずれたときに勝敗バッジと
  絞り込み結果が食い違う**。server は状態を持たず、渡された候補で条件を組み立てるだけにする。
  - **両対局者とも候補に一致する対局は除外する**。`resolveUserSide` が ambiguous として
    勝敗バッジを出さないのと同じ扱いで、バッジと絞り込みの判定を一致させる。
- **並べ替えの基準はセレクトで選ぶ（列ヘッダのクリックにしない）**。既定の「対局日時」は
  `coalesce(playedAt, createdAt)` で、日時列そのものではない（上記の決定）。また「登録日時順」は
  対応する列を持たない。列ヘッダに割り当てると**列と並べ替えキーが 1 対 1 でないことが隠れる**。
- **並びには `id` を副キーとして必ず添える**。同じ日時・同じタイトルの行が 50件境界にまたがると、
  MySQL の順序は不定なのでページ間で行が重複・欠落する。
- **検索語だけ 300ms の debounce**。他は離散的な選択なので即時反映でよいが、テキストは打鍵ごとに
  URL を書き換えると履歴が汚れ、loader も無駄に走る。URL 側が変わったとき（戻る/進む・クリア）は
  入力欄をそちらへ追従させる。
- **検索語の trim は server 側だけで行う**。`validateSearch` で trim すると、入力中の末尾スペースが
  debounce のたびに消えて日本語の分かち書き（「山田 太郎」）が打てなくなる。
- **LIKE のワイルドカードはエスケープする**。`%` を含む対局者名・タイトルを検索したときに
  全件一致になるのを防ぐ（`escapeLike`）。
- **戦型での絞り込みは含めない**。戦型ラベル自体が未実装（[08](../08-roadmap.md)）で、issue #30 の
  範囲外。ここでは対局相手 / 勝敗 / 解析状態 / 期間の 4 つに留める。
