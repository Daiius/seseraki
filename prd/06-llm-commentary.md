# 06. LLM 解説

棋譜と解析結果を整形して LLM に渡すと、質の高い講評が返る。この「貼るだけ」の下準備を提供するのが
本アプリの 3 本目の柱である（[01](./01-domain.md) §1）。データは [05](./05-analysis.md) の解析結果を使う。

---

## 1. 動機

- エンジンは「最善手」と「評価値」は出すが、**なぜ悪手なのか・どう指すべきだったか**は言葉にしない。
- 棋譜 + 評価値推移 + 悪手抽出 + 候補手の読み筋を揃えて LLM に渡すと、これらを踏まえた**日本語の講評**が返る。
- そのため、棋譜詳細から**そのまま貼れる Markdown** を生成する機能を持つ。手動運用（コピペ）から始め、
  将来は自動生成（§3）へ発展させる。

## 2. LLM 解説用 Markdown エクスポート（実装済み）

棋譜詳細画面（[05](./05-analysis.md)）下部の折り畳みセクション「LLM 解説用テキスト」でクリップボードにコピーできる。

### 2.1 生成内容

- **対局情報**（タイトル・対局者・段位・結果・日時）。
- **評価値推移表**（各手の日本語表記 + 先手視点スコア）。
- **注目局面**（悪手 + 評価値変動の top 5）。持ち駒情報も併記。
  - **勝敗が決した局面は除外**（`scoreBefore` の絶対値 ≥ 1000cp。もう挽回不能な局面のぬるい手を注目から外す）。
- **候補手の読み筋**（実手・推奨手・その先の展開の比較構造）。

### 2.2 視点の切り替え

- 設定した自分のユーザー ID から先手/後手を判定し、**ユーザー視点のプロンプト**に切り替える
  （中立視点も可。実装上は `userSide: 'sente' | 'gote' | null`）。
- 結果コード（`SENTE_WIN_CHECKMATE` 等）は日本語化する。

### 2.3 実装方針（プロンプト生成は `shared`、配布は server エンドポイント）

- プロンプト生成は**純関数**（React 非依存・I/O 非依存）で、**`shared` パッケージに置く**
  （入出力は独立した型 `KifuExportInput` / 生成 Markdown。[02](./02-architecture.md) §3.2）。
  盤面追跡・USI→日本語・悪手判定といった依存ロジックも `shared` に集約する（[05](./05-analysis.md)）。
- **server がプロンプト生成エンドポイントを提供する**（例 `GET /kifus/:id/commentary-prompt`）。
  server は `shared` の生成関数を呼んで Markdown を返す。
- **web の「LLM プロンプトをコピー」ボタンはこのエンドポイントを叩く**（自前生成せず、書式を単一真実にする）。
  ※ 現状は web が `packages/web/src/kifu-export/` で自前生成しており、この移設が gap（[08](./08-roadmap.md)）。
- フォーマットの改善は実際に LLM へ貼って講評させたフィードバックに基づく（ローカル限定の非公開メモに記録。`.claude-personal/` 参照）。

## 3. LLM 解説の生成（構想・検証中）

現状は「棋譜詳細から Markdown を手動コピペ → LLM で解説生成」。これを半自動化する。**対象は手動トリガーのみ**
（解析済み全棋譜への自動付与はしない。欲しいものだけ生成し LLM コストを制御下に置く）。

### 3.1 アーキテクチャ（手動トリガー + 軽量な監視スクリプト + server エンドポイント）

- **トリガー**: Web の棋譜詳細の「**解説生成**」ボタン → `commentaries` に `status=queued` の行を作る（[03](./03-data-model.md)）。
  再生成は既存行を `queued` に戻す。
- **commentator = 軽量な監視スクリプト**。**独立した docker container** として worker と同じデスクトップ側で稼働する
  （LLM CLI の実行要件・サブスク認証がそこにあるため。[02](./02-architecture.md) §5）。
- フロー: worker と同じ **outbound polling**（inbound の口を持たない）:
  1. server を polling し、**`status=queued` の解説要求**を claim。
  2. §2.3 の**プロンプト生成エンドポイントを fetch**（commentator 自身は将棋ロジックを持たず薄い）。
  3. **LLM CLI に流して解説を生成**（サブスク内で完結・API 従量課金なし）。
  4. 解説を server API 経由で `commentaries` に書き戻す（`status=done`）。失敗は `status=failed` + `error`。
- 認証は worker と同じ API_KEY 別系統（[07](./07-auth-and-privacy.md)）。
- 必要要素: `commentaries` テーブル（キューを兼ねる。[03](./03-data-model.md)）、`packages/commentator`（薄い loop）、
  server のプロンプト生成エンドポイント + 解説の enqueue/claim/complete エンドポイント。
- **却下**: 解析済み全棋譜への自動付与（コスト・不要な解説が増える。手動トリガーで足りる）/ MCP サーバ経由の
  データ取得（薄いスクリプトには過剰）/ worker への相乗り（解析と解説の責務が混ざる）/ server 内での LLM 実行
  （VPS に CLI 実行環境を要求してしまう）。

### 3.2 Markdown フォーマット改善（先行検討中）

- **戦型ラベル**（初手 5〜7 手のパターンマッチ）。
- SFEN を注目局面に併記、評価値推移表に Δ 列。
- 先手/後手の悪手・好手マーカー分離。
- 冗長区間の圧縮（±50cp 安定区間の省略表記）や、注目局面の絞り込み（ターニングポイント優先）。
