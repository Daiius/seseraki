# 05. 解析と可視化

本章は worker のエンジン解析と、Web 棋譜詳細画面での可視化を定める。
データ型は [03](./03-data-model.md)、解析ドメインの用語は [01](./01-domain.md) §5 を参照。

---

## 1. worker のエンジン解析

worker（`packages/worker`）は server とは分離した実行環境で稼働し、USI プロトコルでやねうら王を駆動する（[02](./02-architecture.md)）。

### 1.1 解析フロー

1. server から**未解析の最古の棋譜を 1 件取得**（`GET /api/worker/kifus`、`usiMoves` と
   **既に入っている局面数（`analyzedCount`）**を含む。[04](./04-ingestion.md)）。
   現行の抽出条件は「`analysisCompletedAt IS NULL` かつ `usiMoves IS NOT NULL`」を **`coalesce(playedAt, createdAt)` 昇順**で 1 件
   （`playedAt` が null の棋譜は `createdAt` で並ぶ。`analysisError IS NULL` も条件に含む＝失敗棋譜は除外。[03](./03-data-model.md)）。
   **最古優先（FIFO）**は「全棋譜を時系列で見る」用途に合わせた選択（[decisions](./_grilling/decisions.md)）。
2. `usiMoves` を使って各局面を **MultiPV（既定 3）で解析**。定跡ヒット時はエンジンが即座に候補手を返す。
   **`analyzedCount` の局面から再開**する（§1.1c）。
3. **1 局面解析するたびに進捗を報告**（`POST /api/worker/analyses/progress`）。§1.1b。
4. 解析結果を **経過時間で区切ったチャンク**として server に送信（`POST /api/worker/analyses`・
   1 チャンク 1 トランザクション）。**完了（`analysisCompletedAt`）は server が件数で確定**する。§1.1c。
5. **ポーリング間隔（既定 10 秒）**で繰り返す。idle 時はログ出力なし。

- KIF→USI 変換は server 側で済んでおり、worker は KIF パーサーを持たない（[01](./01-domain.md) §2 / [04](./04-ingestion.md)）。

### 1.1b 解析の進捗報告（決定・2026-07-21）

本番想定の byoyomi モードでは 1 局の解析に十数分〜数十分かかる。その間 UI から「解析中なのか、キューが
止まっているのか」を区別できないため、進捗を報告して可視化する（表示は §2.5）。

- worker は**局面を 1 つ解析するたびに**進捗（解析済み局面数 / 全局面数 = `usiMoves.length + 1`）を報告する。
  **毎局面**にするのは、N 局面ごとにすると N の適正値が 1 局面あたりの所要時間（MATERIAL/NNUE・
  depth/byoyomi で桁が変わる。§1.2・§1.4）に依存し、固定値の根拠が置けないため。メモリ更新だけで
  DB 書き込みがないので、150 局面 × 数十分に対して最大 150 リクエストは無視できる。
- 報告には `analysisRevision` を添え、**現在の世代と一致しないときは無視する**（submit / error 報告と
  同じ世代照合。reanalyze 後に届いた旧解析の進捗を表示しない）。**完了済み・失敗記録済みの棋譜への報告も
  無視する**（報告と submit が前後しても「終わったのに解析中」が残らない）。
  - ⚠ この照合は DB 読み取り（`await`）を挟むため、**判定してから記録するまでの間に submit / error 報告 /
    reanalyze / 削除が完了しうる**。古い判定のまま書き込むと終わった棋譜の「解析中」が復活し、
    次の報告が無ければ server 再起動まで消えない。よって**メモリへの書き込みはクリアとの
    compare-and-set** にする（クリアのたびに進む単調カウンタを読み取り前に取り、記録時に一致を確かめる）。
- ⚠ **進捗報告の失敗は握りつぶし、解析を継続する。** server が落ちている・遅いときに解析まで止まるのは
  本末転倒。worker は報告の完了を待たずに次の局面へ進む。
- **進捗は server のメモリに持ち、DB に永続化しない**（`packages/server/src/analysis-progress.ts`。
  `swars/job-store.ts` と同じモジュールスコープの状態。単一プロセス前提は [02](./02-architecture.md) §1 /
  [07](./07-auth-and-privacy.md)）。理由: (1) チャンク submit が入れば `moveAnalyses` 件数から導出できる
  ようになるため、列を足すと捨てることになる（migration を 2 回打つ）、(2) 数十分の解析中に DB へ UPDATE を
  繰り返さない、(3) 完了すれば `analysisCompletedAt` が正になる揮発的な情報である。server 再起動で消えるが
  worker の次の報告で復活し、**stale が自動で消える利点**にもなる。よって [03](./03-data-model.md) に列は足さない。
- worker は棋譜を 1 件ずつ処理するので**解析中の棋譜は常に高々 1 件**。エントリは
  **submit 成功 / error 報告 / reanalyze / 棋譜削除でクリア**する。
- 報告先は既存流儀に倣い **`/api/worker/` 配下**に置く（[04](./04-ingestion.md) §7）。
  理由は [`_grilling/decisions.md`](./_grilling/decisions.md)。

### 1.1c 解析結果のチャンク submit と再開（決定・2026-07-21）

全局面を貯めて最後に 1 回だけ送ると、byoyomi 運用で数十分かかる解析が submit の失敗で丸ごと失われる。
チャンクに分けて送り、途中から再開できるようにする。

- **チャンクは経過時間で区切る**（前回 submit から T 秒）。抑えたいのは「失敗時に失われる計算時間」なので、
  時間で切れば**失う上限が T 秒に固定**され、開発 MATERIAL でも本番 NNUE depth20 でも同じ保証になる
  （固定局面数だと同じ N で失う量が桁違いになる。§1.2・§1.4）。T の具体値は実装判断
  （現行は `packages/worker/src/kifu-analysis.ts` の `CHUNK_INTERVAL_MS`）。
- **途中まで入った棋譜は続きから再開する。** `GET /api/worker/kifus` が `moveAnalyses` の件数
  （`analyzedCount`）を返し、worker は `moveNumber = analyzedCount` から解析する。チャンク submit の
  失敗は解析ごと中断する（下記）ので `moveNumber` に穴が空かず、**件数がそのまま再開位置**になる。
  再開がないと、poll 条件は `analysisCompletedAt IS NULL` なので次の poll で同じ棋譜を 0 から
  解析し直し、数十分の計算は結局失われる。
- **`analysisCompletedAt` は server が件数で判定する。** submit のたびに `moveAnalyses` の件数を数え、
  `usiMoves.length + 1` に達したら**同じトランザクション内で**立てる（[03](./03-data-model.md) §3）。
  「揃っていれば完了」という不変条件で決まるため、worker のクラッシュ位置やチャンク境界に依存しない。
  - worker は解析を終えたら**最終チャンクを空でも送る**（再開位置が既に全局面に達しているケースで、
    送らないと「全局面が揃っているのに未完了」の棋譜を拾い直し続ける）。
  - ⚠ 件数で決める以上、**`moveNumber` が棋譜の有効範囲（`0..usiMoves.length`）にあることを
    submit 側で検証する**（外れていれば 1 件も書かずに 400）。範囲外を受け入れると欠番のまま
    件数だけが達して完了扱いになり、poll から外れて自動再開でも直らない（[03](./03-data-model.md) §3）。
- **世代照合（`analysisRevision`）と `FOR UPDATE` による reanalyze との直列化は現行のまま**。
  チャンクごとに世代を確認し、不一致なら破棄する（§1.1a）。
- **冪等性の担保は 3 箇所に分散する**（submit の DELETE を廃止するため）。どこが何を担保するかは
  [03](./03-data-model.md) §3 に書く。
- 進捗（§1.1b）のメモリ保持は**残す**。**メモリ = 表示**（毎局面更新で滑らか）/ **DB 件数 = 再開位置と
  完了判定**（正確）と役割を分ける。DB 件数に一本化すると進捗の粒度が T 秒に落ち、最初の T 秒間は
  「未解析」に見えてしまう。
- 理由は [`_grilling/decisions.md`](./_grilling/decisions.md)。

### 1.1a ポイズンピル対策（解析失敗の扱い）

- **不正棋譜はエンジンを異常終了させうる**（KIF→USI 変換の不備で illegal move になる等）。何もしないと同じ
  kifu を無限に拾い直してクラッシュし続け、キューが詰まる。
- **失敗の分類**:
  - **棋譜起因＝恒久失敗**: illegal move / エンジンプロセスの異常終了 / 解析タイムアウト。当該 kifu の
    `analysisError` に理由（例 `illegal move at move 57`）を記録し（`POST /api/worker/kifus/:id/error`）、
    **エンジンを無条件で再起動**してクリーンな状態に戻してから次の kifu へ進む。error 付き kifu は poll から
    除外される（[03](./03-data-model.md)）。timeout も同じ depth/byoyomi では再発する二次ポイズンピルなので恒久扱い。
  - **インフラ起因＝一時失敗**: server への解析結果 submit 失敗（ネットワーク/DB）等。棋譜自体は正常なので
    `analysisError` は立てず、次の poll で再試行する（エンジン再起動もしない）。
    - ⚠ **チャンク submit の失敗は、その棋譜の解析を中断し、次の poll で続きから再開する**（§1.1c）。
      握りつぶして続行すると `moveNumber` に穴が空き、再開位置を「入っている件数」で決められなくなるため。
      §1.1b の「**進捗報告**の失敗は握りつぶして継続」とは**逆**である点に注意——進捗は表示だけで失っても
      実害がないのに対し、チャンクは計算結果そのものだから。
- worker はエンジンの死（プロセス `close`）を検知したら解析中の待機を即座に打ち切る（60 秒タイムアウトまで待たない）。
- **エンジン再起動が失敗したら**（binary 不在等）、worker は `engineReady = false` にして**復旧できるまで次の kifu を
  取得しない**（dead なエンジンで後続の正常棋譜まで恒久失敗扱いする連鎖を防ぐ）。以降の poll は kifu 取得前に再起動を再試行する。
- worker の報告（submit / error）は取得時の **`analysisRevision` を添える**。server は現在の revision と一致するときだけ
  適用するため、reanalyze でリセットした後に届いた**旧解析の報告は破棄**される（世代競合対策。[03](./03-data-model.md)）。
- UI は「解析失敗」バッジ＋理由を出す。**再試行は `POST /api/kifus/:id/reanalyze`**（[04](./04-ingestion.md) §6）＝
  `kifText` を再変換して `usiMoves`・メタを作り直し、旧 `moveAnalyses` を削除、`analysisError` をクリア、`analysisRevision`
  を +1 して再キューする。これによりパーサ修正が既存の失敗棋譜にも反映される（error クリアだけだと壊れた `usiMoves` のまま再度落ちる）。
- 投入時の整合チェック（[04](./04-ingestion.md) §5）は best-effort で、完全な合法性保証はしない。取りこぼしはここで受ける。

### 1.2 エンジン構成（開発 / 本番）

| | 開発用 (Dockerfile) | 本番用 (Dockerfile.prod) |
|---|---|---|
| EDITION | MATERIAL（駒得。開発環境の低スペック向け軽量版） | NNUE |
| 評価関数 | なし | 評価関数（`nn.bin`, ~60MB） |
| 定跡 | なし | 定跡（ペタブック系） |
| TARGET_CPU | OTHER | OTHER |

> ⚠️ **評価値の精度**: 開発 MATERIAL は駒得ベースのため序盤の評価値が大きめに振れる。
> 本番 NNUE で再解析すると安定する想定（[01](./01-domain.md) §5）。同じ棋譜でもエンジンが変われば数値が変わる。

> ⚠️ **運用注意: エンジン構成（MATERIAL↔NNUE・depth/byoyomi）を変えたら `reanalyze` すること**（決定・2026-07-21）。
> 解析は途中から再開する（§1.1c）ため、構成を変えると 1 つの棋譜に**異なる基準の評価値が混ざりうる**。
> [08](./08-roadmap.md) の確定事項「解析来歴（engine/eval）は持たない」により、再開時に前回のエンジンを
> 比較する手段はない——**検出はせず運用で受ける**（個人用で構成変更はまれ・`reanalyze` は既にある）。

- **本番の解析モードは byoyomi（時間指定）を志向**する（局面の複雑さに応じて深さを自動調整）。ただし**適正値は
  吟味中**で、具体値はリポジトリに固定せず env 運用にとどめる（[decisions](./_grilling/decisions.md)）。

### 1.4 解析 depth の目安

| depth | 用途 | 時間/局面（1CPU） |
|---|---|---|
| 10–12 | 簡易解析 | 数秒 |
| 15–18 | 標準解析 | 数十秒 |
| 20+ | 詳細解析 | 数分 |

## 2. Web 棋譜詳細画面（`/kifus/$id`）

盤面 + 評価値 + 解析結果を並べる。**モバイルファースト縦並び**（sticky グループ → 候補手 →
評価値グラフ → 折り畳み詳細）で、md+ も同順序で `max-width` 制御する。

> **盤面追跡・USI 変換・USI→日本語表記・悪手判定のロジックは `shared` パッケージに置く**（純 TS）。
> web は対話盤面のためクライアント側でこれを import し、server は同じロジックでプロンプトを生成する
> （[02](./02-architecture.md) §3.2 / [06](./06-llm-commentary.md)）。

**局面（`BoardState`）は不変として扱う**。`applyMove` は変化したマス（移動元・移動先）と手番側の持ち駒だけを
差分コピーし、残りの行・駒オブジェクトは元の局面と共有する（1 手ごとに 81 マスをディープコピーしない）。
受け取った側は局面を破壊的に変更してはならない。**全局面は棋譜詳細ページで 1 度だけ構築**し、盤面コンポーネントへ
props で渡す（同一レンダー内で作り直さない）。理由は [`_grilling/decisions.md`](./_grilling/decisions.md)。

### 2.1 盤面と操作

- **将棋盤**: 9x9 テキスト盤面（モバイル `size-8` / md `size-10` / lg `size-12` / xl `size-14` の段階的レスポンシブ。
  マス寸法・グリッドトラック・文字サイズを揃えて拡大）。先手=黒字、後手=赤字 +180 度反転。
  直前の指し手の移動先マスをハイライト（`bg-primary/15`）。**解析前でも `usiMoves` があれば表示可能**。
  後手ユーザーの場合はデフォルト反転。
- **持ち駒表示**: 盤面の上下（反転で入替）。プレイヤー名を左、持ち駒を右。
- **コンパクト情報行**: 直近の指し手 + 局面評価値 + 手数/N + 「分岐」バッジ（分岐中のみ）を 1 行に集約（`flex-wrap`）。
- **コントローラー**: ⏮ ◀ slider ▶ ⏭ + 盤面反転（SVG アイコン）。モバイルは slider 非表示・◀▶ を大型化、
  md+ は `btn-sm` + slider。
- **キーボード操作**（決定・2026-07-21）: `←` / `→` で 1 手戻る / 進む、`Home` / `End` で最初 / 最後へ。
  **分岐中（§2.2）の `←` / `→` は分岐内を移動**し、分岐の先頭で `←` すると本筋へ復帰する。`Home` / `End` は
  分岐を解除して本筋の端へ飛ぶ（コントローラーの ⏮ ⏭ と同じ）。入力欄（slider・メモ等）にフォーカスがある間と
  修飾キー併用時はブラウザ既定に譲る。理由は [`_grilling/decisions.md`](./_grilling/decisions.md)。
- **盤面の左右タップ**（決定・2026-07-21）: 盤面の右半分をタップで 1 手進む / 左半分で 1 手戻る（モバイルで
  コントローラーまで指を運ばずに並べるため。全画面幅で有効）。左右の判定は**画面基準**で、盤面反転の影響を
  受けない。挙動は ◀ ▶ ボタンと機械的に同じ＝**分岐中のタップは本筋へ復帰**する（キーボードの `←` / `→` は
  分岐内を移動するので非対称。キーボードは読み筋を辿る道具、ボタンとタップは本筋を送る道具という住み分け）。
  端では何もしない（開いている候補手 `details` を無用に閉じないため）。理由は
  [`_grilling/decisions.md`](./_grilling/decisions.md)。
- **sticky 化**: 盤面 + コンパクト情報行 + コントローラーを `sticky top-0` で上端固定し、スクロール中も操作可能。

### 2.2 評価値と候補手

- **局面評価値**: 実手後の局面のスコアを **先手視点へ正規化**して表示（保存値は手番視点。moveNumber の parity で
  符号反転する `toSenteEval`。[01](./01-domain.md) §5 / [03](./03-data-model.md)）+ 形勢判断ラベル（互角/有利/優勢/勝勢/詰み）。
- **候補手一覧**: 各候補を `<details name="candidates">` の排他アコーディオンで展開。
  展開時に読み筋 + 探索深さ + 実手マーク + 最善手との差異 + **分岐操作 ◀▶** を表示。
  本筋を進めると ref 経由で開いている details を自動クローズ。初期 3 件、4 件以上は chevron で折り畳み。
  md+ は `cursor-default` で常時表示ベース。
- **日本語表記**: USI→駒名付き日本語変換（例 `▲７六歩(77)`）。盤面追跡で駒名を解決（[01](./01-domain.md) §2）。

### 2.3 悪手判定

- **判定条件**: 局面評価値が手番側にとって **300cp 以上悪化**し、**かつ実手が候補手リスト外**の場合に悪手（[01](./01-domain.md) §5）。
- 表示: 悪手 = 赤い ▲/△（手番対応）、非悪手の非最善手 = ※（warning 色）。出す場所は
  **盤面直下の候補手一覧**（§2.2）と、棋譜全体を俯瞰する**評価値グラフの悪手マーカー**（§2.4）。

### 2.4 評価値グラフ

- visx（`@visx/responsive` `@visx/scale` `@visx/shape`）による SVG 折れ線グラフ。先手有利 = 上、後手有利 = 下。
  塗りは評価値 0 の中央線を基準に上下へ描く。
- 幅は `ParentSize` でコンテナ幅に追従（実寸描画で歪みなし）。非常に長い棋譜のみ下限幅まで広げ横スクロール。
- コントローラー / コンパクト行と連動、クリックで局面移動。現在手位置に縦線。
- **悪手マーカー**（先手 ▲ / 後手 ▽、相手の悪手は透明度を下げる）。
- モバイルは高さ 180px・余白詰めでナビゲーション用途を強化、デスクトップは 120px。

### 2.5 一覧・ヘッダー・折り畳み

- **棋譜一覧（`/`）**: テーブル表示。解析済み/未バッジ、勝敗バッジ（自分の名前候補 `VITE_SELF_NAMES` ∪ `VITE_SWARS_USER_ID` に一致時）、
  swars 取り込みの「更新」ボタン（[04](./04-ingestion.md)）。サーバー未接続時は警告。既定は対局日時の新しい順・ページネーション付き。
  - **日時列は「対局日時」だが、`playedAt` が無い棋譜は登録日時（`createdAt`）で代替する**（一覧の並びも
    `coalesce(playedAt, createdAt)` 降順・[04](./04-ingestion.md) §6）。代替表示のときは
    **「（登録）」を添えて弱色にする**（決定・2026-07-21）。理由は [`_grilling/decisions.md`](./_grilling/decisions.md)。
  - **検索・絞り込み・並べ替えは server 側で行い、条件は URL のクエリに持つ**（決定・2026-07-21）。
    テーブル上部に、検索語（タイトル・対局者名）/ 解析状態 / 勝敗 / 期間 / 並べ替えの基準・向き /
    「条件をクリア」を並べる。パラメータは [04](./04-ingestion.md) §6.1 と同じ。
    - **フィルタバーは「検索」の折り畳み（`<details>`）に収め、既定は閉じたままにする**（暫定の決定・2026-07-21）。
      現状の絞り込み軸だけでは常時出しておくほど使わず、一覧の上部を占有するため。
      **条件が効いていても自動では開かず、効いている条件を畳んだままの `summary` に 1 行で出す**
      （条件付きの URL やリロードで「なぜ件数が少ないのか」が読めないと困る一方、自動で開くと
      占有を減らす目的と衝突するため）。**ヒット件数は折り畳みの外**に置き、閉じていても見えるようにする。
      中身は入力要素が数個で軽く、開閉のたびに状態を作り直したくないので
      **遅延マウント（本節「ページ下部の折り畳み」の決定）は使わない**。
      戦型を記録できるようになれば常時表示へ戻す判断があり得る。理由は [`_grilling/decisions.md`](./_grilling/decisions.md)。
    - 検索語だけ 300ms の debounce でドラフトを URL に反映する（打鍵ごとに履歴を積まない）。
      長さの上限（100字）は入力欄の `maxLength` と URL の正規化の両方で頭打ちにする
      （超えた値を送ると server が 400 を返し、一覧全体がエラー表示になるため）。
    - **解析中の棋譜は「未」バッジの代わりに「解析中 N/M」+ 最終更新からの経過を出す**（決定・2026-07-21）。
      2 値の「解析中」だけでは worker がハングしても「解析中」のままで、「解析中なのか、キューが
      止まっているのか」を半分しか解決しない。**進捗が動くこと自体が生存確認になる。**
      - ⚠ **stale は閾値で消さない。** §1.3 のとおり byoyomi の適正値は吟味中で env 運用であり、
        開発 MATERIAL と本番 NNUE で 1 局面あたりの所要時間が桁違いになるため、「何分更新が無ければ
        死んでいる」の閾値に根拠が置けない（depth 20+ なら 1 局面数分かかり、正常な解析を stale と
        誤判定しうる）。**経過時間を出して判断は人に委ねる。**
      - 取得は専用の **`GET /api/analysis/progress`**（[04](./04-ingestion.md) §6）。解析中は高々 1 件なので、
        一覧も詳細もこれ 1 つを見て自分の id と一致したら表示する。**一覧 SQL を再実行せずに進捗だけ
        ポーリングできる。**
      - ⚠ **状態フィルタ（`all` / `analyzed` / `unanalyzed` / `failed`）に「解析中」は追加しない。**
        進捗はメモリにあり SQL で絞り込めないため、「絞り込みは server 側（SQL）で行い、件数・ページングも
        同じ条件で数える」（[04](./04-ingestion.md) §6.1）を崩すことになる。
      - **解析中は短間隔、非解析中は長間隔でポーリング**する（進捗エンドポイントはメモリ参照だけで
        DB を触らないので安い）。進捗エントリが消えた＝完了/失敗なので、そのタイミングで
        ルーターのローダーを作り直す（一覧のバッジ・詳細の解析結果はローダー経由のため、
        作り直さないと「解析中」が消えた後に古い「未」のまま残る）。
    - **勝敗のセレクトは自分の名前候補が設定されているときだけ出す**（判定できないため）。
    - 絞り込みを変えたら `page` を 1 に戻す。0 件のときの案内は絞り込みの有無で出し分ける。
    - 理由は [`_grilling/decisions.md`](./_grilling/decisions.md)。
- **詳細ヘッダー**: 「← 一覧」+ タイトル + ⋯ ドロップダウン（再解析・削除を格納し誤操作防止）。
  - **削除・再解析は結果を必ず画面に返す**（決定・2026-07-21）。失敗（HTTP エラー / 接続不能）は
    ページ上部の `alert-error`、再解析の受理は `alert-info` で通知し、実行中は両ボタンを無効化する。
    削除は成功すると一覧へ遷移するため通知しない。理由は [`_grilling/decisions.md`](./_grilling/decisions.md)。
  - **解析中はページ上部に「解析中 N/M・◯前に更新」+ 進捗バー**を出す（決定・2026-07-21）。一覧と同じ
    `GET /api/analysis/progress` を見て、`kifuId` が一致したときだけ表示する（§2.5 の一覧バッジと同源）。
  - **解析中は部分結果がそのまま見える**（決定・2026-07-21）。チャンク submit（§1.1c）で途中まで入った
    解析結果が、評価値グラフ・候補手・悪手判定にそのまま反映される（解析が進んでいく様子が見えるのは
    歓迎する副次効果）。**UI 側の特別な対応は入れない**——消費者はいずれも部分結果で壊れない
    （`detectBlunders` は連続ペアのみ判定・`EvalGraph` は 2 点未満で描画しない・`kifu-export` は
    入っている局面を回すだけ）。解析中の棋譜から LLM 解説用テキストをコピーすると途中までの内容に
    なるが、**無効化も警告もしない**（途中までの解説が欲しい場面もありうる）。
- **ページ下部の折り畳み**（`<details>`・既定閉じ）: KIF テキスト /
  LLM 解説用テキスト（[06](./06-llm-commentary.md)）/ メモ。
  - **開かれるまで中身をマウントしない**（決定・2026-07-21）。`<details>` は閉じていても中身がレンダーされるため、
    エクスポート用テキスト生成のような重い処理が閉じたまま毎レンダー走ってしまう。一度開いた後はマウントしたままにする
    （開閉の往復を軽くし、メモの入力中の下書きも失わない）。理由は [`_grilling/decisions.md`](./_grilling/decisions.md)。
  - **全局面を 1 行ずつ並べたテーブルは持たない**（決定・2026-07-21）。全局面 × 全候補手の「候補手詳細」に加え、
    全局面 × 最善手 1 件の「局面評価値」テーブルも置かない。読み筋・評価値の閲覧は盤面直下の候補手一覧と
    コンパクト情報行（§2.2）を局面移動しながら使い、棋譜全体を俯瞰する用途は評価値グラフ + 悪手マーカー
    （§2.4）と LLM 解説用テキスト（[06](./06-llm-commentary.md)）が担う。理由は
    [`_grilling/decisions.md`](./_grilling/decisions.md)。
