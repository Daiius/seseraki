# 05. 解析と可視化

本章は worker のエンジン解析と、Web 棋譜詳細画面での可視化を定める。
データ型は [03](./03-data-model.md)、解析ドメインの用語は [01](./01-domain.md) §5 を参照。

---

## 1. worker のエンジン解析

worker（`packages/worker`）は server とは分離した実行環境で稼働し、USI プロトコルでやねうら王を駆動する（[02](./02-architecture.md)）。

### 1.1 解析フロー

1. server から**未解析の最古の棋譜を 1 件取得**（`GET /api/worker/kifus`、`usiMoves` を含む。[04](./04-ingestion.md)）。
   現行の抽出条件は「`analysisCompletedAt IS NULL` かつ `usiMoves IS NOT NULL`」を **`coalesce(playedAt, createdAt)` 昇順**で 1 件
   （`playedAt` が null の棋譜は `createdAt` で並ぶ。`analysisError IS NULL` も条件に含む＝失敗棋譜は除外。[03](./03-data-model.md)）。
   **最古優先（FIFO）**は「全棋譜を時系列で見る」用途に合わせた選択（[decisions](./_grilling/decisions.md)）。
2. `usiMoves` を使って各局面を **MultiPV（既定 3）で解析**。定跡ヒット時はエンジンが即座に候補手を返す。
3. 解析結果を server に**トランザクションで送信**（`POST /api/worker/analyses`）。
4. **ポーリング間隔（既定 10 秒）**で繰り返す。idle 時はログ出力なし。

- KIF→USI 変換は server 側で済んでおり、worker は KIF パーサーを持たない（[01](./01-domain.md) §2 / [04](./04-ingestion.md)）。

### 1.1a ポイズンピル対策（解析失敗の扱い）

- **不正棋譜はエンジンを異常終了させうる**（KIF→USI 変換の不備で illegal move になる等）。何もしないと同じ
  kifu を無限に拾い直してクラッシュし続け、キューが詰まる。
- **失敗の分類**:
  - **棋譜起因＝恒久失敗**: illegal move / エンジンプロセスの異常終了 / 解析タイムアウト。当該 kifu の
    `analysisError` に理由（例 `illegal move at move 57`）を記録し（`POST /api/worker/kifus/:id/error`）、
    **エンジンを無条件で再起動**してクリーンな状態に戻してから次の kifu へ進む。error 付き kifu は poll から
    除外される（[03](./03-data-model.md)）。timeout も同じ depth/byoyomi では再発する二次ポイズンピルなので恒久扱い。
  - **インフラ起因＝一時失敗**: server への解析結果 submit 失敗（ネットワーク/DB）等。棋譜自体は正常なので
    `analysisError` は立てず、次の poll で再試行する（エンジン再起動もしない）。
- worker はエンジンの死（プロセス `close`）を検知したら解析中の待機を即座に打ち切る（60 秒タイムアウトまで待たない）。
- **エンジン再起動が失敗したら**（binary 不在等）、worker は `engineReady = false` にして**復旧できるまで次の kifu を
  取得しない**（dead なエンジンで後続の正常棋譜まで恒久失敗扱いする連鎖を防ぐ）。以降の poll は kifu 取得前に再起動を再試行する。
- worker の報告（submit / error）は取得時の **`analysisRevision` を添える**。server は現在の revision と一致するときだけ
  適用するため、reanalyze でリセットした後に届いた**旧解析の報告は破棄**される（世代競合対策。[03](./03-data-model.md)）。
- UI は「解析失敗」バッジ＋理由を出す。**再試行は `POST /api/kifus/:id/reanalyze`**（[04](./04-ingestion.md) §6）＝
  `kifText` を再変換して `usiMoves`・メタを作り直し、旧 `moveAnalyses` を削除、`analysisError` をクリア、`analysisRevision`
  を +1 して再キューする。これによりパーサ修正が既存の失敗棋譜にも反映される（error クリアだけだと壊れた `usiMoves` のまま再度落ちる）。
- 投入時の整合チェック（[04](./04-ingestion.md) §5）は best-effort で、完全な合法性保証はしない。取りこぼしはここで受ける。

### 1.2 エンジン構成（開発 / 本番）

| | 開発用 (Dockerfile) | 本番用 (Dockerfile.prod) |
|---|---|---|
| EDITION | MATERIAL（駒得。開発環境の低スペック向け軽量版） | NNUE |
| 評価関数 | なし | 評価関数（`nn.bin`, ~60MB） |
| 定跡 | なし | 定跡（ペタブック系） |
| TARGET_CPU | OTHER | OTHER |

> ⚠️ **評価値の精度**: 開発 MATERIAL は駒得ベースのため序盤の評価値が大きめに振れる。
> 本番 NNUE で再解析すると安定する想定（[01](./01-domain.md) §5）。同じ棋譜でもエンジンが変われば数値が変わる。

### 1.3 エンジンオプション

| オプション | 環境変数 | 既定 | 説明 |
|---|---|---|---|
| Threads | `ENGINE_THREADS` | 1 | |
| USI_Hash | `ENGINE_HASH` | 128 (MB) | 本番は 2048–4096 推奨 |
| MultiPV | `ENGINE_MULTIPV` | 3 | 候補手数 |
| — | `ENGINE_DEPTH` | 10 | 探索深さ |
| — | `ENGINE_BYOYOMI` | 未設定 | 秒読み(ms)。設定時は depth より優先（局面の複雑さで深さを自動調整） |
| EvalDir | `ENGINE_EVAL_DIR` | — | 本番: 評価関数ディレクトリ |
| BookDir | `ENGINE_BOOK_DIR` | — | 本番: 定跡ディレクトリ |

- 定跡関連の細目（BookFile / IgnoreBookPly / FlippedBook / BookOnTheFly / BookMoves / BookEvalDiff /
  BookDepthLimit 等）は BookDir 設定時に本番向けの値へ自動設定される（メモリ節約・回転局面ヒット・末端採用など）。
- **本番の解析モードは byoyomi（時間指定）を志向**する（局面の複雑さに応じて深さを自動調整）。ただし**適正値は
  吟味中**で、具体値はリポジトリに固定せず env 運用にとどめる（[decisions](./_grilling/decisions.md)）。

### 1.4 解析 depth の目安

| depth | 用途 | 時間/局面（1CPU） |
|---|---|---|
| 10–12 | 簡易解析 | 数秒 |
| 15–18 | 標準解析 | 数十秒 |
| 20+ | 詳細解析 | 数分 |

## 2. Web 棋譜詳細画面（`/kifus/$id`）

盤面 + 評価値 + 解析結果を並べる。**モバイルファースト縦並び**（sticky グループ → 候補手 →
評価値グラフ → 折り畳み詳細）で、md+ も同順序で `max-width` 制御する。

> **盤面追跡・USI 変換・USI→日本語表記・悪手判定のロジックは `shared` パッケージに置く**（純 TS）。
> web は対話盤面のためクライアント側でこれを import し、server は同じロジックでプロンプトを生成する
> （[02](./02-architecture.md) §3.2 / [06](./06-llm-commentary.md)）。

**局面（`BoardState`）は不変として扱う**。`applyMove` は変化したマス（移動元・移動先）と手番側の持ち駒だけを
差分コピーし、残りの行・駒オブジェクトは元の局面と共有する（1 手ごとに 81 マスをディープコピーしない）。
受け取った側は局面を破壊的に変更してはならない。**全局面は棋譜詳細ページで 1 度だけ構築**し、盤面コンポーネントと
折り畳みの表へ props で配る（同一レンダー内で作り直さない）。理由は [`_grilling/decisions.md`](./_grilling/decisions.md)。

### 2.1 盤面と操作

- **将棋盤**: 9x9 テキスト盤面（モバイル `size-8` / md `size-10` / lg `size-12` / xl `size-14` の段階的レスポンシブ。
  マス寸法・グリッドトラック・文字サイズを揃えて拡大）。先手=黒字、後手=赤字 +180 度反転。
  直前の指し手の移動先マスをハイライト（`bg-primary/15`）。**解析前でも `usiMoves` があれば表示可能**。
  後手ユーザーの場合はデフォルト反転。
- **持ち駒表示**: 盤面の上下（反転で入替）。プレイヤー名を左、持ち駒を右。
- **コンパクト情報行**: 直近の指し手 + 局面評価値 + 手数/N + 「分岐」バッジ（分岐中のみ）を 1 行に集約（`flex-wrap`）。
- **コントローラー**: ⏮ ◀ slider ▶ ⏭ + 盤面反転（SVG アイコン）。モバイルは slider 非表示・◀▶ を大型化、
  md+ は `btn-sm` + slider。
- **キーボード操作**（決定・2026-07-21）: `←` / `→` で 1 手戻る / 進む、`Home` / `End` で最初 / 最後へ。
  **分岐中（§2.2）の `←` / `→` は分岐内を移動**し、分岐の先頭で `←` すると本筋へ復帰する。`Home` / `End` は
  分岐を解除して本筋の端へ飛ぶ（コントローラーの ⏮ ⏭ と同じ）。入力欄（slider・メモ等）にフォーカスがある間と
  修飾キー併用時はブラウザ既定に譲る。理由は [`_grilling/decisions.md`](./_grilling/decisions.md)。
- **sticky 化**: 盤面 + コンパクト情報行 + コントローラーを `sticky top-0` で上端固定し、スクロール中も操作可能。

### 2.2 評価値と候補手

- **局面評価値**: 実手後の局面のスコアを **先手視点へ正規化**して表示（保存値は手番視点。moveNumber の parity で
  符号反転する `toSenteEval`。[01](./01-domain.md) §5 / [03](./03-data-model.md)）+ 形勢判断ラベル（互角/有利/優勢/勝勢/詰み）。
- **候補手一覧**: 各候補を `<details name="candidates">` の排他アコーディオンで展開。
  展開時に読み筋 + 探索深さ + 実手マーク + 最善手との差異 + **分岐操作 ◀▶** を表示。
  本筋を進めると ref 経由で開いている details を自動クローズ。初期 3 件、4 件以上は chevron で折り畳み。
  md+ は `cursor-default` で常時表示ベース。
- **日本語表記**: USI→駒名付き日本語変換（例 `▲７六歩(77)`）。盤面追跡で駒名を解決（[01](./01-domain.md) §2）。

### 2.3 悪手判定

- **判定条件**: 局面評価値が手番側にとって **300cp 以上悪化**し、**かつ実手が候補手リスト外**の場合に悪手（[01](./01-domain.md) §5）。
- 表示: 悪手 = 赤い ▲/△（手番対応）、非悪手の非最善手 = ※（warning 色）。出す場所は
  **盤面直下の候補手一覧**（§2.2）と、棋譜全体を俯瞰する**評価値グラフの悪手マーカー**（§2.4）。

### 2.4 評価値グラフ

- visx（`@visx/responsive` `@visx/scale` `@visx/shape`）による SVG 折れ線グラフ。先手有利 = 上、後手有利 = 下。
  塗りは評価値 0 の中央線を基準に上下へ描く。
- 幅は `ParentSize` でコンテナ幅に追従（実寸描画で歪みなし）。非常に長い棋譜のみ下限幅まで広げ横スクロール。
- コントローラー / コンパクト行と連動、クリックで局面移動。現在手位置に縦線。
- **悪手マーカー**（先手 ▲ / 後手 ▽、相手の悪手は透明度を下げる）。
- モバイルは高さ 180px・余白詰めでナビゲーション用途を強化、デスクトップは 120px。

### 2.5 一覧・ヘッダー・折り畳み

- **棋譜一覧（`/`）**: テーブル表示。解析済み/未バッジ、勝敗バッジ（自分の名前候補 `VITE_SELF_NAMES` ∪ `VITE_SWARS_USER_ID` に一致時）、
  swars 取り込みの「更新」ボタン（[04](./04-ingestion.md)）。サーバー未接続時は警告。既定は対局日時の新しい順・ページネーション付き。
  - **日時列は「対局日時」だが、`playedAt` が無い棋譜は登録日時（`createdAt`）で代替する**（一覧の並びも
    `coalesce(playedAt, createdAt)` 降順・[04](./04-ingestion.md) §6）。代替表示のときは
    **「（登録）」を添えて弱色にする**（決定・2026-07-21）。理由は [`_grilling/decisions.md`](./_grilling/decisions.md)。
  - **検索・絞り込み・並べ替えは server 側で行い、条件は URL のクエリに持つ**（決定・2026-07-21）。
    テーブル上部の 1 行に、検索語（タイトル・対局者名）/ 解析状態 / 勝敗 / 期間 / 並べ替えの基準・向き /
    「条件をクリア」/ ヒット件数を並べる。パラメータは [04](./04-ingestion.md) §6.1 と同じ。
    - 検索語だけ 300ms の debounce でドラフトを URL に反映する（打鍵ごとに履歴を積まない）。
      長さの上限（100字）は入力欄の `maxLength` と URL の正規化の両方で頭打ちにする
      （超えた値を送ると server が 400 を返し、一覧全体がエラー表示になるため）。
    - **勝敗のセレクトは自分の名前候補が設定されているときだけ出す**（判定できないため）。
    - 絞り込みを変えたら `page` を 1 に戻す。0 件のときの案内は絞り込みの有無で出し分ける。
    - 理由は [`_grilling/decisions.md`](./_grilling/decisions.md)。
- **詳細ヘッダー**: 「← 一覧」+ タイトル + ⋯ ドロップダウン（再解析・削除を格納し誤操作防止）。
  - **削除・再解析は結果を必ず画面に返す**（決定・2026-07-21）。失敗（HTTP エラー / 接続不能）は
    ページ上部の `alert-error`、再解析の受理は `alert-info` で通知し、実行中は両ボタンを無効化する。
    削除は成功すると一覧へ遷移するため通知しない。理由は [`_grilling/decisions.md`](./_grilling/decisions.md)。
- **ページ下部の折り畳み**（`<details>`・既定閉じ）: KIF テキスト /
  LLM 解説用テキスト（[06](./06-llm-commentary.md)）/ メモ。
  - **開かれるまで中身をマウントしない**（決定・2026-07-21）。`<details>` は閉じていても中身がレンダーされるため、
    エクスポート用テキスト生成のような重い処理が閉じたまま毎レンダー走ってしまう。一度開いた後はマウントしたままにする
    （開閉の往復を軽くし、メモの入力中の下書きも失わない）。理由は [`_grilling/decisions.md`](./_grilling/decisions.md)。
  - **全局面を 1 行ずつ並べたテーブルは持たない**（決定・2026-07-21）。全局面 × 全候補手の「候補手詳細」に加え、
    全局面 × 最善手 1 件の「局面評価値」テーブルも置かない。読み筋・評価値の閲覧は盤面直下の候補手一覧と
    コンパクト情報行（§2.2）を局面移動しながら使い、棋譜全体を俯瞰する用途は評価値グラフ + 悪手マーカー
    （§2.4）と LLM 解説用テキスト（[06](./06-llm-commentary.md)）が担う。理由は
    [`_grilling/decisions.md`](./_grilling/decisions.md)。

## 3. 未確認・将来の論点

- **局面単位の再解析**: 特定局面だけ depth を変えて再解析（Web の「この局面を深く解析」ボタン →
  worker に解析リクエスト）。優先度: 低。再解析結果は**その局面の候補手を最新 1 世代に上書き**する
  （depth 別の世代は持たない。[03](./03-data-model.md) 決定済み）。局面ごとに depth が混在しても
  `candidateMoves.depth` で判別できる（[08](./08-roadmap.md)）。
- 本番 NNUE での再解析による評価値の安定化（開発 MATERIAL 由来の序盤ブレ対策。§1.2）。
- **解析エンジン・評価関数の来歴は DB に持たない**（決定。§1.2 の二重性は本番 NNUE 単一運用で受ける。[03](./03-data-model.md)）。
