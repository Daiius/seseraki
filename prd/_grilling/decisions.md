## grill-me 決定ログ

### 時制方針: 「あるべき設計の理想形」を基準にする（2026-07-16）
- PRD は現実装の記録に留めず、**あるべき設計の理想形を正**として書く。
- 現実装が理想と異なる箇所は、理想を本文に書いた上で**現状との gap** として明示・管理する。
- 帰結: 現 DB スキーマ・アーキが理想でない場合、PRD は理想を記述し、実装追従は gap 項目にする。
- 前提は変わらず**個人用・ローカル運用**。理想は「個人用棋譜解析ツールとしての理想」であって
  エンタープライズ化ではない（過剰設計を避ける）。

### 解析来歴（engine/eval）は持たない（2026-07-16）
- **決定**: 解析エンジン・評価関数の識別子（MATERIAL/NNUE・評価関数名）を DB に**記録しない**。
- 理由: 本番は NNUE 単一運用で開発 MATERIAL の数値は一時的な確認用。異常な評価値は目視で気づける。
  エンジン側に自己申告を仕込むのは手間で、個人用途には**過剰設計**。
- 保持するのは `candidateMoves.depth`（候補手単位。既存）のみ。
- **帰結**: 局面単位の再解析は depth 別の複数世代を持たず、**最新 1 世代に上書き**（単一エンジン前提）。

### LLM 解説の生成アーキ（2026-07-16）
- **決定**: プロンプト（解析結果込み Markdown）生成を **server のエンドポイント化**（例 `GET /kifus/:id/commentary-prompt`）。
- **決定（対象範囲）**: **手動トリガーのみ**（解析済み全棋譜への自動付与はしない）。Web の「解説生成」ボタンで
  キュー投入 → commentator が拾って生成。欲しいものだけ生成し LLM コストを制御下に置く。
- **commentator = 軽量な監視スクリプト**（独立 docker container・worker 側デスクトップに配置）:
  **キュー（`commentaries.status = 'queued'`）を polling** → プロンプトを server から fetch → LLM CLI に流す →
  解説を `commentaries` へ書き戻す。commentator 自身は将棋ロジックを持たず薄く保つ。サブスク内で完結し API 従量課金なし。
- web の「LLM プロンプトをコピー」ボタンも同じプロンプトエンドポイントを使い、書式を単一真実にする。

### shared パッケージを作る（旧「作らない」を改定）（2026-07-16）
- **決定**: `packages/shared` を新設する。
- **責務（凝集。junk drawer にしない）**:
  1. 将棋ドメインの**純ロジック**（board 盤面追跡・USI 変換・USI→日本語・悪手判定・kifu-export プロンプト生成）。
  2. **zod 検証スキーマ**（runtime バリデーション）。
- **旧判断の再評価**: 「shared を作らない」は元々「Hono RPC の型共有を使わず zod 定義を shared に溜め込む」
  傾向への予防だった。しかし (1) 対話盤面ロジックを web と server（プロンプト生成）で共有する必要が生じ、
  (2) **型共有だけでは動作時に不正データを弾けず、runtime 検証には zod の実体が要る**。→ shared の意義が明確化。
- **維持する原則**: **API レスポンス/リクエストの型共有は引き続き Hono RPC**。shared に API 型を重複させない。
- **帰結**: board/usi/kifu-export を web→shared へ移設。server は shared の kifu-export でプロンプト生成
  エンドポイントを提供。commentator は薄いまま。web は対話盤面で shared を import。
- KIF/CSA パーサーは現状 server 専用（消費者が server のみ）なので shared に移さない（第2の消費者が出たら再検討）。

### commentaries テーブル（2026-07-16）
- **決定**: `commentaries` は kifu と **1:1**、再生成は**同行を上書き**（解析来歴非保持の方針と揃える）。
- **キューを兼ねる**（別途キューテーブルは持たない。薄い watcher 思想）: `status`(queued/done/failed) を持ち、
  「解説生成」ボタンで `queued` 行を作る/戻す → commentator が `queued` を polling → 生成後 `done`。
  失敗は `failed` + `error` で受け、無限リトライを防ぐ（poison-pill と同じ思想）。
- カラム: `kifuId`(PK/FK), `status`(enum), `body`(text?, 完了まで null), `llmModel`(varchar?),
  `promptVersion`(varchar?), `error`(text?, failed 用), `createdAt` / `updatedAt`。
- Web の「解説あり」表示は `status = 'done'` で判定。

### 入力形式のスコープ（2026-07-16）
- **決定**: Web 手動投入は **KIF 貼り付け（実装済み）+ CSA 直接貼り付け**（既存の CSA→KIF 変換器を再利用）。
- **対象外**: KIF ファイルアップロード / SFEN（単一局面で棋譜解析のスコープ外）。
- CSA→KIF 変換器は現状は取り込み専用ディレクトリ配下だが、CSA 貼り付けは一括取り込みと無関係なので、
  一般的な変換として server 内の中立な場所に置く（消費者は server のみなので shared には移さない）。

### ポイズンピル対策（解析できない棋譜）（2026-07-16）
- **主軸 = worker 側の失敗状態**: kifus に `analysisError`(nullable) を追加。worker はエンジンの
  異常終了/illegal move を検知したら当該 kifu に記録し、**エンジンを再起動して次へ**進む。
  poll は「`analysisCompletedAt IS NULL` かつ `analysisError IS NULL` の最古」を引く。
  → ポイズンピルはスキップされ**キューが詰まらない**。UI に「解析失敗」バッジ、再試行は error クリア。
- **投入時検証は best-effort（保証ではない）**: 完全な合法性検証は将棋ルール一式（打ち歩詰め等）の
  実装が要り容易でない。`applyMove` が破綻しないか程度の軽い整合チェックに留め、取りこぼしは worker 側で受ける。
- 具体的なバグの症状・再現・修正手順といった**作業メモは PRD に書かず** `.claude-personal/` に置く。

### KIF 変換の堅牢化・メタ抽出・失敗時動作（2026-07-17）

実棋譜（`８四全` を含む 70 手）で「記録は入るが解析エンジンが illegal move で異常終了し、以後同じ棋譜を
無限リトライしてキューが停止する」不具合を起点に、入力面と失敗時動作をまとめて詰めた。

- **原因**: KIF パーサが成銀の一文字略記 `全`（同様に `圭`=成桂 / `杏`=成香）を未対応で手 57 が黙って欠落 →
  1 手欠落で手番 parity がズレ、構文上妥当な USI 列のまま盤面が破綻 → illegal move。加えて `kifToUsiMoves` が
  パースエラーを無視して部分的 `usiMoves` を保存していた。
- **入力側の決定**:
  - パースエラー（指し手行の変換失敗）または**非平手手合割**なら `usiMoves = null` で保存（記録・メタは残す）。
    壊れた部分 `usiMoves` を worker に渡さないのが要点。完全合法性チェック（`shared` + `applyMove`）は将来の理想として残し、
    当面はこの軽量ガード＋ worker 失敗状態で受ける（best-effort の具体化）。
  - 成駒略記対応・終局マーカー認識（`詰み` 等・最初のマーカーで指し手収集を打ち切り＝末尾の分岐ブロック混入防止）。
  - メタ抽出（sente/gote/dan/playedAt）。**result は終局マーカー＋手番 parity から swars 互換コードを導出**し、
    既存の勝敗バッジ・LLM エクスポートを無改修で活かす。
  - **タイトル任意化・自動生成**（空なら `${sente} vs ${gote}`）。旧「KIF はユーザー入力」を改定（KIF インポート常用化のため）。
- **失敗時動作の決定**:
  - 失敗分類: 棋譜起因（illegal / エンジン異常終了 / **timeout**）＝恒久（`analysisError` 記録 → poll 除外）、
    インフラ起因（submit 失敗）＝一時（再試行）。恒久時は**エンジン無条件再起動**。
  - `analyze()` はプロセス `close` で即 reject（60 秒ハング解消）。
  - 再試行 UI アクション＝**`POST /kifus/:id/reanalyze`**: `kifText` 再変換で `usiMoves`・メタを作り直し
    `analysisError`/`analysisCompletedAt` をクリアして再キュー（`title`/`memo` は温存）。error クリアだけだと
    壊れた `usiMoves` のまま再度落ちるため、再変換で既存の壊れ棋譜も復旧する。
- **スコープ外（今回入れない）**: `shared` 抽出を伴う完全合法性チェック、KIF ファイルアップロード、駒落ちの USI 変換対応。
- 具体的なバグ症状・再現・修正手順の作業メモは `.claude-personal/`（公開 PRD には書かない）。

### 棋譜詳細の「候補手詳細」テーブルを廃止（2026-07-21）

棋譜詳細ページ下部の折り畳み「候補手詳細」（全局面 × 全候補手のテーブル）を**製品仕様として持たない**と決めた
（[05](../05-analysis.md) §2.5 を改訂。issue #26 / PR #36）。

- **理由（運用）**: 実運用でまず開いていない。読み筋を見たい局面は評価値グラフの悪手マーカーから飛べるので、
  その局面の候補手一覧（[05](../05-analysis.md) §2.2・読み筋 + 探索深さ + 実手マーク + 分岐操作）で用が足りる。
- **理由（コスト）**: `<details>` は閉じていても中身がレンダーされる。この表は局面数 × MultiPV 行を生成し、
  各行の読み筋日本語化が手数ぶん `applyMove`（盤面のディープコピー）を回すため、**詳細ページを開いた瞬間に
  走る盤面コピーの最大の発生源**だった。遅延レンダーや仮想化で覆う手もあるが、使っていない機能のために
  実装を増やす判断はしない。
- **[01](../01-domain.md) §1-2「疑問手と分岐を一望したい」との整合**: この動機は評価値グラフ + 悪手マーカー
  （棋譜全体の俯瞰）と、局面ごとの候補手一覧（深掘り）の組み合わせで満たす。加えて全局面ぶんの
  候補手・評価値をテキストで一覧したい場合は LLM 解説用テキスト（[06](../06-llm-commentary.md)）が同等の情報を持つ。
  よって 01 の改訂は不要とした。
- **データ側は不変**: `candidateMoves.pv` は盤面直下の候補手表示・kifu-export・悪手判定が使い続ける。
  スキーマ・API・worker に変更なし（表示先が 1 つ減るだけ）。
