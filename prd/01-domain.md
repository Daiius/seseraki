# 01. ドメインとプロダクト動機

このアプリが扱う対象は **将棋の棋譜** と、その **エンジン解析結果** である。
データモデル（[03](./03-data-model.md)）・投入（[04](./04-ingestion.md)）・解析（[05](./05-analysis.md)）は、
すべてこの章のドメイン事実を地盤とする。実装エージェントはまずここを参照すること。

---

## 1. プロダクトの動機（なぜ作るか）

個人が自分の対局を振り返るための道具である。市販・無料の将棋ソフトでも解析はできるが、次の 3 点を
「自分の運用に合う形」で束ねたい、というのが出発点:

1. **棋譜を溜め込んで自動で解析したい。** 対局のたびに手動で解析させるのは面倒なので、対局履歴から
   まとめて取り込み、対局相手・勝敗・日時を起こしてタイトルを付け、あとは放っておけばデスクトップ PC が
   順番に解析してくれる、という状態を作る（[04](./04-ingestion.md) / [05](./05-analysis.md)）。
2. **疑問手と分岐を一望したい。** 「どこで形勢を損ねたか」「その局面で最善は何だったか」を、評価値グラフ・
   悪手マーカー・候補手の読み筋として並べて眺める（[05](./05-analysis.md)）。
3. **LLM に講評させたい。** 棋譜と解析結果を整形して LLM に渡すと、意外と的を射た分析が返る。
   その「貼るだけ」の下準備をアプリ側で用意する（[06](./06-llm-commentary.md)）。

将棋ソフトとの相互運用も視野に入れ、**KIF 形式の棋譜を読める**ようにしておく（貼り付け。ファイルアップロードは対象外・§6）。
これにより取り込み経路によらず、いろいろな出所の棋譜が同じパイプラインに乗る。

## 2. 棋譜フォーマット（KIF / CSA / USI）

本アプリは複数の棋譜表現を扱い、内部処理は **USI** に寄せる。

| 形式 | 役割 | 扱い |
|---|---|---|
| **KIF** | 人間可読な標準形式。ユーザーの入力窓口 | 原本を `kifText` に保管。登録時に server が USI へ変換 |
| **CSA** | swars 取り込み時の中間形式 | swars 取り込みフローで CSA→KIF 変換してから登録（[04](./04-ingestion.md)） |
| **USI** | エンジン・内部処理用の指し手表記 | `usiMoves`(string[]) として保持。worker と Web 盤面が消費 |

- **KIF→USI 変換は server 側で棋譜登録時に一度だけ実行する**（`packages/server/src/kif`）。
  worker は KIF パーサーを持たず、`usiMoves` だけを受け取って解析する（[05](./05-analysis.md)）。
- 日本語の指し手表記（例 `▲７六歩(77)`）は Web 側で USI から盤面追跡で復元する（[05](./05-analysis.md) §日本語表記）。

## 3. 対局のメタ情報

1 局の棋譜からは、指し手列のほかに次のメタが取れる（取れないものは nullable）:

| 項目 | 説明 |
|---|---|
| `sente` / `gote` | 先手 / 後手プレイヤー名 |
| `senteDan` / `goteDan` | 先手 / 後手の段位（取得元にあれば） |
| `result` | 対局結果（勝敗・決着の種類） |
| `playedAt` | 対局日時 |

- これらは対局メタであり、**一括取り込み・KIF 貼り付けの両経路とも登録時に抽出して `kifus` に格納する**
  （KIF 経路はヘッダ〔先手/後手/段位/開始日時〕を読み、`result` は終局マーカー＋手番 parity から導出。[04](./04-ingestion.md) §3）。
- **対局タイトルの自動生成**に使う: 「誰と対戦したか」を人間可読なタイトルに起こす（KIF 貼り付けは `title` 任意で、
  空なら `${sente} vs ${gote}` を自動生成）。これがあると一覧画面（[05](./05-analysis.md)）で対局を見分けられる。
- Web 一覧では、自分の名前候補（`VITE_SELF_NAMES` のカンマ区切り ∪ `VITE_SWARS_USER_ID`）のいずれかが
  対局者名に一致すれば、そのユーザー視点で**勝敗バッジ**を出す。将棋アプリごとに自分の表示名が異なる
  （例: `Daiius` / `daiius`）ため複数候補を許容する。両対局者とも候補に一致した場合は側を確定できない
  ため自分視点表示は無効化し、詳細画面で警告を出す。

## 4. 棋譜の供給源

- 棋譜の主な入り口は 2 つ: **swars からの一括取り込み**と **KIF 貼り付け**（[04](./04-ingestion.md)）。
- swars 一括取り込みは複数対局をまとめて登録できる。**swars の正式名称・取得の詳細な仕組み・アクセス姿勢・資格情報**は
  公開文書に書かず、`.claude-personal/` の運用メモに集約する（[README](./README.md) §公開リポジトリでの秘匿方針）。
- どちらの入り口でも **KIF→USI 変換・対局メタ抽出・解析の下流は共通**（メタ抽出は両経路とも実装済み。§3 / [04](./04-ingestion.md) §3）。

## 5. 解析ドメイン（評価値・候補手・疑問手・分岐）

エンジン解析が生む概念。用語と数値の意味を固定する（実装は [05](./05-analysis.md)）。

- **局面（moveNumber）**: `moveNumber = N` は **N 手適用後・N+1 手目を指す前の局面**（`0` は初期局面）。
  **偶数 = 先手番 / 奇数 = 後手番**（その局面で指す側）。1 棋譜 = 局面の列。
- **評価値（score）**: **保存値は USI エンジンが返す手番視点のスコア**（正規化せず `candidateMoves` にそのまま格納。[03](./03-data-model.md)）。
  型は `cp`（centipawn）または `mate`（詰みまでの手数）。表示・判定時に **moveNumber の parity で先手視点へ正規化**する
  （後手番＝奇数の局面は符号反転。実装 `toSenteEval`。[05](./05-analysis.md)）。形勢判断ラベル（互角/有利/優勢/勝勢/詰み）は先手視点値に写像。
- **候補手（MultiPV）**: 各局面でエンジンが返す上位 N 手（既定 3）。`rank=1` が最善。
  各候補は指し手・スコア・**読み筋（pv）**・探索深さ（depth）を持つ。
- **CPL（centipawn loss）**: **最善手の評価 − 実手の評価**（手番側視点）。実手の評価は、**実手が候補手に
  含まれていればその `scoreValue`、含まれていなければ次局面の最善値を符号反転**して得る（後者は近似。
  [05](./05-analysis.md) §2.3）。同一局面・同一手番の比較なので先手視点への正規化は要らない。
- **疑問手 / 悪手**: 実際に指された手が最善から大きく外れた手。判定は CPL の 2 段階
  ——**悪手**（CPL ≧ 悪手閾値）/ **疑問手**（疑問手閾値 ≦ CPL < 悪手閾値）。**「好手」は設けない**
  （相手が損したのは相手の悪手であって自分の好手ではなく、真の好手は MultiPV=3 では判定できない）。
  閾値は UI から変更でき、既定は悪手 300cp / 疑問手 150cp（[05](./05-analysis.md) §2.3・§2.5）。
- **詰み系（詰み逃し / 詰まされ）**: **mate が絡む変化は CPL を計算せず別カテゴリとして扱う。**
  詰みは連続量ではなく、cp に写像すると 30000 級の損失になって平均 CPL を 1 局で壊すため。
- **決着**: 局面の最善評価値の絶対値が決着閾値（既定 1000cp）以上なら勝負が決したとみなし、
  **ラベルを付けない**（CPL の計算自体はする）。詰み系は cp の量ではないのでこの対象外。
- **分岐**: ある局面で実手の代わりに候補手を選んで読み筋を辿る操作。「もしこう指していたら」を Web で再生できる。

> ⚠️ 評価値は評価関数に依存する。開発環境の MATERIAL（駒得ベース）は序盤で大きめに振れ、
> 本番の NNUE（評価関数同梱）の方が安定する（[05](./05-analysis.md) §エンジン構成）。同じ棋譜でも
> 解析エンジンが変われば数値が変わることを前提にする。

## 6. 未確認・将来の論点

- ✅ **入力形式のスコープ決定**: Web 手動投入は **KIF 貼り付け + CSA 直接貼り付け**（[04](./04-ingestion.md) §2）。
  KIF ファイルアップロード・SFEN は**対象外**（SFEN は単一局面で棋譜解析のスコープ外）。
- 対局結果コード（`SENTE_WIN_CHECKMATE` 等）の網羅性と日本語化辞書の維持（[06](./06-llm-commentary.md) が使用）。
- 段位・棋力レートなどメタ項目の拡充（分析軸として使うか）。
